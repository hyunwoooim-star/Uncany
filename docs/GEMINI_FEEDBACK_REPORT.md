## 1. 기술적 피드백 요약

### 1.1 동시성 제어 (Advisory Lock)

| 평가 | 내용 |
|------|------|
| ✅ 긍정 | Advisory Lock + Exclusion Constraint 조합 "탁월한 선택" |
| ✅ 긍정 | DB 레벨에서 물리적 중복 차단 → 데이터 무결성 높음 |
| ⚠️ 주의 | Timeout 설정 점검 필요 (학기 초 예약 몰림 대비) |
| ⚠️ 주의 | Deadlock, 세션 타임아웃 처리 검증 필요 |
| 💡 제안 | Lock 대기 시간이 UX 해치지 않는지 확인 |

### 1.2 RLS 보안

| 평가 | 내용 |
|------|------|
| ✅ 긍정 | RLS 정책 설계 견고함 |
| 💡 제안 | '승인된 교사(approved)' 상태 체크하는 Custom SQL Function 추가 |
| 💡 제안 | 단순 Authenticated → RBAC(권한 기반 제어)로 강화 |
| 💡 제안 | RLS 복잡해지면 Materialized View 또는 Custom Claims 활용 고려 |

### 1.3 Edge Functions

| 항목 | 피드백 |
|------|--------|
| Cold Start | 번들링 크기 최소화, 자주 사용하는 학교 정보 캐싱 권장 |
| 취소 정책 | '10분 전 취소' 서버 단에서 한 번 더 검증 필수 |
| 알림 시스템 | Push 알림/알림톡 연동 Edge Functions에서 처리 권장 |
| NEIS API | CORS 문제, 환경 변수(Secret) 관리 확인 필요 |

### 1.4 Flutter Web 최적화

| 항목 | 피드백 |
|------|--------|
| 렌더링 모드 | html vs canvaskit - 학교 PC 사양 낮을 수 있으므로 최적화 필요 |
| Deep Linking | `/schedule/:date/:room_id` 같은 URL 구조 설계 확인 |
| 초기 로딩 | CanvasKit 사용 시 한글 폰트 렌더링 최적화 필요 |
| Responsive | 태블릿/PC 가로 너비에 따른 레이아웃 대응 선행 필요 |

### 1.5 Soft Delete

| 피드백 |
|--------|
| deleted_at 활용 → 통계 데이터(교실 활용도 분석) 유리 |
| 모든 Query에 필터 누락 방지 → PostgreSQL View 생성 또는 RLS에 `deleted_at IS NULL` 기본 포함 |

## 2. UX/기능 피드백 요약

### 2.1 교시 기반 시스템

| 피드백 |
|--------|
| 학교마다 교시 시간(40분/45분/50분) 다름 → **학교별 맞춤형 시간표 설정** 기능 필요 |
| A/B 주간 등 복잡한 시간표 직관적 반영 여부 확인 |

### 2.2 관리자 기능

| 제안 |
|------|
| **자동 승인 허용 교실** vs **승인 필수 교실** 구분 옵션 |
| 전체 예약 현황 한눈에 보기 + **엑셀 내보내기** 기능 |
| 관리자 강제 취소 기능 (행사 등 특수 상황 대응) |

### 2.3 노쇼(No-show) 관리

| 피드백 |
|--------|
| 교실 사용 직후 다음 교시 교사 바로 들어옴 → 노쇼 관리 기능 고려 |

### 2.4 모바일 대응

| 제안 |
|------|
| 교사들 이동 중 예약 확인 필요 → **PWA** 설정 권장 |
| 별도 앱 설치 없이 홈 화면 추가로 모바일 앱처럼 사용 유도 |

### 2.5 NEIS API 확장

| 제안 |
|------|
| 학사 일정(공휴일, 개교기념일, 재량휴업일) API 연동 |
| 쉬는 날 예약 자동 차단 또는 경고 |
